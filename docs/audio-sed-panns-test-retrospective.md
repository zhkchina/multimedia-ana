## 试验目标

目标是为下游“二创卡点视频”链路提供基础音频事件检测服务，聚焦：

- 打斗
- 枪击
- 爆炸

并输出适合后续检索和与视频分析结果求交的事件时间轴数据。

## 实施内容

试验期间新增了一个独立的 `audio-sed` 双容器服务：

- `audio-sed-api`
- `audio-sed-worker`

推理模型使用 `PANNs Cnn14_DecisionLevelMax`，输入支持视频和音频文件，服务会：

- 探测多条音轨
- 逐音轨提取音频
- 输出逐音轨事件结果

## 已确认有效的部分

以下部分在工程上已经验证可行：

- API/worker 双容器模式可复用现有工程风格
- worker 可按需拉起并支持空闲退出
- 多音轨逐条分析与逐条返回是正确方向
- 输出结构可收敛成适合 `jsonl` 和数据库索引的事件对象
- 带音轨的短视频片段可以稳定跑通端到端流程

## 核心问题

### 1. 模型对目标业务的区分能力不足

本次最关键的问题不是服务接线，而是检测效果。

在两个用于对照的短片段中：

- `19:34-19:46` 片段包含打斗场面
- `19:46-20:00` 片段不应视为明显打斗命中

在多轮调试后，服务仍只在前者的一条非默认音轨上检出极少量事件，且召回明显不足。

这说明：

- 模型对电影混音下的动作声学特征不够敏感
- 以当前标签集合和后处理策略，无法稳定覆盖“多处打斗动作点”

### 2. 降低阈值没有带来实质性召回提升

测试中尝试过将阈值从 `0.15` 下调到：

- `0.12`
- `0.10`
- `0.08`

结果没有显著新增有效事件。

这表明问题不主要在阈值，而在：

- 模型原始分数分布本身不理想
- 当前标签与电影动作声的匹配度不足
- 后处理之前的模型信号就已经偏弱

### 3. 原始标签虽可输出，但业务命中仍然稀疏

试验后期已经将输出收紧为：

- 仅保留过阈值事件
- 每条音轨只返回命中的标签、领域和事件

但即使这样，真正命中的原始标签仍然很少，典型仅有：

- `Whack, thwack`

这与“片段里有多处打斗动作”不一致。

也就是说，问题不是 JSON 设计，而是检测结果本身不够用。

### 4. 默认音轨未必是最有效音轨

测试中出现过：

- 默认音轨几乎无命中
- 非默认音轨才出现有效分数

这验证了“必须逐音轨返回，不能默认混轨或只看默认轨”的判断。

但也同时说明：

- 下游如果依赖单一默认音轨，会漏检
- 基础服务需要把音轨问题暴露给下游，而不是服务端替业务做简化判断

### 5. 不适合直接作为当前路线继续投入

本次实验说明，`PANNs + 现有标签映射 + 当前后处理` 这条路线：

- 可以做验证
- 可以做参考原型
- 但不适合作为当前要继续推进的主方案

主要原因不是部署，而是效果上限和业务适配度。

## 过程中踩到的具体坑

### 无音轨测试片段

早期用于 smoke test 的若干 `scene_0350 ~ scene_0359` 文件没有音频流，导致：

- worker 无法抽音轨
- 无法判断模型效果

结论：

- 冒烟片段必须先校验音频流
- 测试素材生成流程需要强制保留音轨

### 整片输入不适合当前推理方式

把整部电影直接送入 PANNs 推理会导致显存峰值过高。

结论：

- 长视频必须切片或滑窗
- 整片直推不是合理使用方式

### `top_labels` 对当前业务帮助有限

虽然 `top_labels` 可用于观察分数分布，但对业务判断没有决定性价值。

下游真正关心的是：

- 过阈值事件有没有出现
- 何时开始
- 何时峰值
- 何时结束
- 强度是多少

所以后续更合适的输出应该继续围绕事件本身，而不是摘要分数。

## 结论

本次试验的主要结论是：

- 工程接法基本可行
- 多音轨逐轨分析方向正确
- JSON 结构可以设计得足够适合下游索引
- 但 `PANNs` 在本业务上的实际召回不够，不能满足当前需求

因此，本次 `audio-sed` 代码不继续留在 `main`，只保留：

- 归档分支
- `/data` 下测试产物
- 本复盘文档

## 后续建议

后续新技术栈选择时，优先关注以下能力：

- 面向电影/短视频动作场景的更强事件定位能力
- 对打斗、枪击、爆炸等短瞬态事件更高召回
- 更适合逐事件时间轴输出，而不是仅做弱分类
- 能与多音轨、scene 切片、视频侧动作判断形成自然组合
